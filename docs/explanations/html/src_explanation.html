NestJS Enterprise - Technical Explanation

Triết lý Hệ thống

Hệ thống được thiết kế theo mô hình Clean Architecture kết hợp với các nguyên lý Enterprise:

  • Transaction Isolation: Đảm bảo tính toàn vẹn dữ liệu cho các thao tác tài chính.
  • Optimistic/Pessimistic Caching: Sử dụng Redis để giảm tải database.
  • Unified Response: Chuẩn hóa dữ liệu trả về qua Interceptors.

Sơ đồ Hệ Thống Tổng Quan

Module Auth: Bảo Mật & Xác Thực

Sử dụng JWT (JSON Web Token) với cơ chế Double Token (Access & Refresh). Access Token được lưu trong memory/cookie thời hạn ngắn (15m), Refresh Token được hash và lưu trong DB (7d) cùng HttpOnly Cookie.

AuthService.login POST /auth/login 
graph TD
A[Bắt đầu] --> B{Kiểm tra User}
B -- Không thấy --> C[Lỗi 401]
B -- Thấy --> D{So khớp Pass}
D -- Sai --> C
D -- Đúng --> E[Tạo JWT Tokens]
E --> F[Hash Refresh Token]
F --> G[Lưu DB & Set Cookie]
G --> H[Trả về Access Token]
IN: LoginDto { email, password }
OUT: AuthResponseDto { success, accessToken, user }
Xác thực danh tính qua email và bcrypt. Nếu thành công, hệ thống tạo cặp token, ghi đè hash của Refresh Token vào database của người dùng đó để quản lý phiên làm việc.

Module Catalog: Hiệu Năng Đọc

Module này được tối ưu hóa cho tốc độ đọc. Mọi yêu cầu đều đi qua Redis trước khi truy cập Database.

ProductsService.findProducts GET /catalog/products 
graph LR
A[Yêu cầu] --> B[Tạo Cache Key]
B --> C{Redis có data?}
C -- Có --> D[Trả về ngay]
C -- Không --> E[Query Prisma]
E --> F[Lưu Redis]
F --> D
IN: ProductQueryDto { page, limit, category, search, sort... }
OUT: { data: Product[], total: number }
Sử dụng getOrSet pattern. Cache key được tạo động dựa trên tất cả query parameters để đảm bảo phân trang và lọc dữ liệu chính xác.

Module Orders: Thanh Toán & Transaction

Sử dụng Prisma Transaction để đảm bảo quy trình thanh toán diễn ra nguyên tử (Atomic). Nếu bất kỳ bước nào lỗi, toàn bộ thay đổi sẽ được rollback.

OrdersService.checkout POST /orders/checkout 
graph TD
subgraph Transaction
A[Lấy Giỏ Hàng] --> B[Kiểm tra Tồn Kho]
B --> C[Tạo Đơn Hàng]
C --> D[Trừ Tồn Kho]
D --> E[Đóng Giỏ Hàng]
end
E --> F[Trả về Order]
IN: CheckoutDto { address }
OUT: OrderResponseDto
Quy trình nghiêm ngặt: Kiểm tra trạng thái sản phẩm (ACTIVE/Deleted) -> Kiểm tra số lượng tồn kho -> Khấu trừ tồn kho (sử dụng decrement của Prisma).

Module Admin: Quản Trị & Invalidation

Quản lý dữ liệu hệ thống. Quan trọng nhất là cơ chế Cache Invalidation: khi Admin thay đổi thông tin sản phẩm, cache tương ứng phải bị xóa ngay lập tức.

AdminProductsService.updateProduct PATCH /admin/products/:id 
graph TD
A[Yêu cầu Admin] --> B[Verify Quyền]
B --> C[Update Database]
C --> D[Xóa Cache Sản phẩm]
D --> E[Xóa Cache Danh sách]
E --> F[Trả về Kết quả]
IN: UpdateProductDto
OUT: Product Entity
Khi update, hệ thống xóa cả cache chi tiết (theo slug cũ và mới) lẫn cache danh sách sản phẩm để đảm bảo khách hàng luôn thấy dữ liệu mới nhất.
